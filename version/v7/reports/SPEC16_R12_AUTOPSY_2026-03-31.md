# Spec16 R12 Autopsy

- date: `2026-03-31`
- frozen raw baseline: `spec16_scene_bundle_l3_d192_h384_ctx768_r9`
- comparison rung: `spec16_scene_bundle_l3_d192_h384_ctx768_r11`
- blocked pilot: `spec16_scene_bundle_l3_d192_h384_ctx768_r12`

## Verdict

`r12` was not a near-miss pilot. It was a global output-contract collapse on a brittle repair-heavy curriculum, and deterministic repair could not recover exactness.

## What Changed

- `r12` kept the same unique pretrain and midtrain row sets as `r11`.
- The differences between `r11` and `r12` datasets were row ordering, not row membership.
- `r12` reduced actual processed training compute to roughly one-third of `r11`.

Evidence:

- pretrain row-set equality: `True`
- midtrain row-set equality: `True`
- pretrain same-position rate: `1220 / 1330` (`91.7%`)
- midtrain same-position rate: `1144 / 1235` (`92.6%`)
- `r11` pretrain actual processed tokens: `2,571,264`
- `r12` pretrain actual processed tokens: `852,480`
- `r11` midtrain actual processed tokens: `2,260,224`
- `r12` midtrain actual processed tokens: `753,408`

## Failure Signature

The failures are semantic, not just syntactic.

- raw exact on `r12`: `0.0%`
- repaired exact on `r12`: `0.0%`
- raw renderable on `r12`: `35.4%`
- repaired renderable on `r12`: `43.8%`
- exact improvements from deterministic repair: `0`
- renderable-only improvements from deterministic repair: `4`

Common observed error modes:

- mixed-family tag soup
- duplicated `[bundle]`
- copied prompt/control markers like `[OUT]` and `[task:svg]`
- missing stop marker on `15` cases
- non-empty tail text on `26` cases
- missing required singleton fields such as `form`, `theme`, `density`, `tone`
- wrong-family topology keys such as `links` on memory-map bundles and `brackets/cards` on system-diagram bundles

Representative `r12` render errors:

- `missing form`
- `missing theme`
- `invalid integer for arrows: None`
- `invalid integer for links: None`
- `unexpected topology keys for memory_map: links`
- `unexpected topology keys for system_diagram: brackets, cards`
- `unsupported form 'linear_pipeline' for family 'timeline'`

## Diagnosis

The `r11/r12` repair-heavy prompt surface is brittle. On that surface, a reduced-compute pilot is not a safe or informative proxy for the full branch.

The key lesson is not just "pilot budgets are risky." It is:

- same brittle row set
- same repair-heavy instruction surface
- lower compute

was enough to collapse all three families simultaneously.

That means this curriculum is too unstable to use as the default substrate for more raw repair rungs.

## Operational Lessons

- Do not use `r11/r12`-style repair prose as the default raw-training surface.
- Do not treat a same-row-set reduced-compute pilot as a new branch.
- Do not approve new raw training just because the pilot loss looks healthy.
- Treat `r12` as evidence that the current unresolved tail is no longer a standard CE repair-rung problem.

## Conditions Before Future Training Is Allowed

- Keep `r9` frozen as the raw winner.
- Stay on `r9 + deterministic repair` as the default path.
- No new raw `spec16` rung may be another narrow repair-prose continuation.
- The next allowed raw branch must be a clean capacity branch on the frozen `r9` recipe.
- Any future pilot must declare both selected stage budgets and expected actual processed tokens, and compare pilot fraction against the frozen baseline on actual processed tokens.
- Any future pilot must improve `system_diagram` without any family regression or hidden regression relative to `r9`.
- If a candidate branch reuses the same unique row set as `r12`, treat it as a compute/order experiment only, not as a new training idea.
