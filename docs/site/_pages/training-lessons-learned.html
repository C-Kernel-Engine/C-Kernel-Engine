The Arc: Spec by Spec

This is the full trajectory of the visual-DSL training line. Each spec changed the learning contract; each set of rungs explored that contract’s ceiling.

Early Line (Spec02–03): Raw SVG + Tokenizer Discovery

The project started trying to teach a tiny model to emit raw SVG. Spec03’s tokenizer audit revealed the foundational problem: 0 out of 50 canonical control tags were captured as exact BPE tokens. The model was fighting to learn both the spelling of the DSL and the semantics simultaneously.

Lesson: Promote canonical control tags to reserved/protected tokens. Treat interface atoms explicitly. The model should learn scene decisions, not subword spelling.

Spec09–10: The Scene DSL Revolution

The biggest design pivot. Backward-designed from real shipped SVG assets instead of synthetic grids. Defined 7 layout families, 20+ reusable components, and the foundational split:

model → scene DSL → compiler → SVG

The model would learn visual planning, not SVG rendering. This mirrors what CodeT5 and phi-1 argue: structured formal output requires a different representation than free text.

This is where the real learning happened. Three families: table_matrix, decision_tree, memory_map.

For this narrow bounded contract, curriculum shape — balance, ordering, repair ratio — was the dominant lever, not capacity. However, this does not prove capacity is irrelevant for the harder bounded-intent routing problem that spec17/18 attempted. Spec13: The Generalization Wall

  • spec13a: Tried intent prompts (topic+goal+audience → inferred layout). The model collapsed to a single attractor — one family for everything. This is the memorization-vs-generalization barrier.
  • spec13b: Graph-family work. Discovered the model was learning asset labels inside the DSL, not transferable structure. Led to simplifying output contracts and the domain-agnostic principle.

Spec14a: The Default-Board Trap

R1 scored 19.2% exact but 92.3% renderable. The model learned one default board (infra_dark/amber/rings/3-metrics/2-callouts) and replayed it for every prompt. Root cause: block-ordered corpus with 48 identical rows in sequence. The model shortcut to memorizing the most frequent pattern.

Dataset forensics showed: 21/21 non-exact rows predicted [tone:amber] when 12 required blue and 2 required green. The model learned the family shell but not the variant-control tokens.

Spec14b–15: Domain-Agnostic Single Families

The right insight: strip ALL topic/domain info from model-facing prompts. Individual families each got stable within their narrow contracts:

  • spec15a R9 memory_map — decode-validated
  • spec15b R2 system_diagram — decode-validated
  • spec14b R3 timeline — decode-validated

Hypothesis: teach family+form routing first, delay style/topology. R1 result: 4.76% exact, 0% held-out. Worse than spec17 R4. Curriculum decomposition alone was not sufficient at this model size and data mass.

The Pattern: Why “Progress in Rungs” Keeps Failing

The recurring cycle across every spec:

New spec → early learning → narrow repair attempts → fix one thing, break another → hit ceiling → new spec → reset to near-zero → repeat

This is not random bad luck. It is a structural problem with the rung-repair approach to training.

The 7 Root Causes

1. The Narrow Repair Trap

Adding repair rows to fix specific failures (missing [/scene], wrong family) introduces noise that degrades previously-solved slices. Under cross-entropy, repair rows are just more positive targets — the model doesn’t know they were “corrections.”

Evidence: Spec12 R4 — more repair pressure → renderability collapsed from 90.9% to 45.5%. Spec16 R12 — repair-heavy curriculum collapsed entirely under reduced compute.

2. Cross-Family Interference

Strengthening one family weakens others. Each family competes for the same model capacity under standard cross-entropy training.

Evidence: Spec12 R6 — memory_map went from 5/11 to 10/11, but decision_tree crashed from 11/11 to 3/11.

3. The Generalization Wall

Within every spec, memorizing seen prompts is easy (often 100% train exact). But held-out generalization stays near zero when the task requires compositional reasoning. The model learns specific input→output mappings, not the underlying visual planning language.

Evidence: Spec17 R4 and Spec18 R1 — both 0% held-out exact despite clean process and scaffolds. Spec14a R1 — one default board replayed for all prompts.

4. Measurement Bugs Masquerade as Training Failures

Throughout the history: probe pipeline bugs, blueprint/catalog misalignment, seed integrity issues, decode budget problems, rendered_svg_ok: null despite correct output. Multiple rungs were wasted diagnosing infrastructure problems misidentified as training failures.

Evidence: Spec14a R1 — probe adapter not registered. Spec17 R2 — audit validated the blueprint, not the generated catalog, so missing surfaces went undetected.

5. Too Many Axes Change Between Specs

Despite the methodology explicitly forbidding this, each new spec typically changes tokenizer + DSL grammar + prompt surface + curriculum + evaluation criteria simultaneously. This makes it impossible to attribute improvement or regression to any single cause.

Evidence: Spec14a R2 had to be reclassified as a “reset line” because it changed tokenizer, prompt surface, curriculum, and dataset ordering all at once.

6. Canary Budget Miscalibration

Underpowered canaries can catch catastrophic collapse but cannot validate learning, especially for bridge/routing tasks that need multiple passes through the curriculum.

Evidence: Spec17 R2 — ran 1/3 of one epoch per stage. Spec16 R12 — ran 1/3 of R11’s compute. Both produced misleading results.

7. The Semantic Gap Between Explicit and Intent Prompts

Explicit control prompts work well (spec12 R17: 100%, spec16 R9: 81.25%). Intent prompts (topic+goal+audience → visual plan) are a fundamentally harder task requiring compositional generalization the tiny model hasn’t achieved. This may be a genuine capacity/data-quality threshold, not just a curriculum-shape problem.

The Deepest Failure: Bundle-Emission Threshold

Post-review correction: The original analysis framed spec17/18 failures as “wrong family choice” — the model routing to the wrong family. That understates the problem. On held-out spec17, spec18, and spec19 rows, the model is often not emitting a parseable bundle family/form at all. The failure is upstream of misrouting.

This is a critical distinction:

Failure LevelWhat It Looks LikeWhat It Means
Bundle-emission failure Output is prompt garbage, partial fragments, missing [bundle], or no parseable family/form The model has not generalized the bundle-emission contract itself to unseen prompt surfaces. The generalization threshold for structured output under novel prompts has not been crossed.
Family misrouting Output is a valid bundle but with the wrong family or form for the intent The model has generalized bundle emission but not the intent→family mapping. This would be a routing problem.
Style/topology drift Output has the right family/form but wrong theme, tone, or counts Routing works, fine-grained control does not. This is the easiest to fix.

The spec17/18/19 evidence shows most held-out failures are at the first level, not the second or third. The model can emit valid bundles for seen training prompts (some train exact > 0) but cannot transfer the bundle-emission pattern to novel prompt surfaces at all. This is a generalization threshold problem, and it means:

  • Curriculum decomposition alone (spec18’s routing-first approach) is not enough if the model cannot even emit the container structure on novel inputs
  • The fix requires either a much larger and more diverse training corpus (so more prompt surfaces are “seen”) or more model capacity (so the structural pattern generalizes from fewer examples) or both
  • This is why the compiler-as-data-factory approach is so important: it can generate the volume and diversity of clean rows needed to push the emission threshold below the holdout surface

What the Research Says

Each paper maps directly to a problem observed in the spec history.

Chinchilla: You’re Probably Token-Starved

Optimal token count scales linearly with parameter count at roughly 20 tokens per parameter. The narrow spec datasets may be undertrained in total tokens. More importantly, Chinchilla says compute should be split evenly between model size and data tokens. CKE retrains from scratch on each new spec rather than accumulating clean tokens.

Applied to CKE: Compute packed token budgets and effective epochs per spec. Stop copying old totals into new DSLs. Accumulate rather than reset.

Hoffmann et al., 2022 — Training Compute-Optimal Large Language Models

Grokking: Train Long — But Only After Clean Data

On small algorithmic datasets, models first memorize, then much later (sometimes 100x the memorization compute) suddenly generalize. Weight decay is the key lever — it penalizes memorized solutions and forces the model toward the simpler generalizing solution.

Applied to CKE: The pattern of running a few rungs and then changing the spec when generalization doesn’t emerge may be exactly wrong. The model might need to train much longer on a stable contract with strong weight decay. However, “train longer” is a hypothesis to test after the corpus is large and clean via compiler-generated data, not a prescription to run longer on the current small hand-mapped datasets. Grokking on a noisy 90-row corpus would just be grokking noise.

Power et al., 2022 — Grokking: Generalization Beyond Overfitting on Small Algorithmic Datasets

phi-1: Data Quality Is Your Biggest Lever

A 1.3B model trained on 7GB of “textbook-style” code data beat models trained on 50x more tokens. The key: dense, structured, pedagogically ordered data. CKE’s SVG domain is perfect for this — you can programmatically generate structurally valid, maximally diverse training data.

Applied to CKE: Use the compiler backward to generate thousands of valid scene bundles programmatically with controlled variation. Every row compiler-verified. No hand-mapping. Repair/hygiene rows can still exist for closure and stop-boundary teaching, but they should be a small capped fraction, not the curriculum backbone.

Gunasekar et al., 2023 — Textbooks Are All You Need

Codex/HumanEval: pass@k as Diagnostic, Greedy as Gate

pass@k: generate k samples and check if any are correct. CKE’s greedy exact-match at temperature=0 is the harshest possible evaluation. For structured output with a deterministic compiler, generating 5-10 candidates and picking the one that compiles correctly is a powerful inference tactic and diagnostic tool.

Applied to CKE: The compiler is a structural filter. Use pass@k as a side analysis to understand model capability and as a practical inference strategy. However, promotion gates should stay on greedy held-out exactness. If a model only succeeds via sampling luck, it is hiding training weakness, not solving the problem. pass@k tells you “how close is the model to working?” — greedy exact tells you “does the model actually know this?”

Chen et al., 2021 — Evaluating Large Language Models Trained on Code

CodeT5: Identifier-Aware Pre-Training

Code identifiers need specific pre-training signals beyond next-token prediction. CKE’s DSL tokens ([layout:poster_stack], [theme:infra_dark], [tone:amber]) are exactly like code identifiers — they have constrained semantics.

Applied to CKE: Consider pre-training objectives that specifically teach relationships between DSL tokens: masked scene-attribute prediction, contrastive objectives between valid/invalid attribute combinations, or denoising objectives that corrupt one scene field and train the model to restore it.

Wang et al., 2021 — CodeT5: Identifier-aware Unified Pre-trained Encoder-Decoder Models

AlphaCode: Generate-Then-Filter

Generate many candidates, filter by structural validity. Combined with CKE’s deterministic compiler, this is directly applicable: generate 10-50 candidate bundles, compile each one, score by structural validity + visual similarity, keep the best.

Applied to CKE: Turns the model from “must be exact” to “must be roughly right,” which is a much easier task. The compiler does the precision work.

Li et al., 2022 — Competition-Level Code Generation with AlphaCode

NVIDIA Nemotron: Systematic Data Curation

Nemotron’s approach: quality-score every row with a lightweight model (filter out low quality), generate synthetic variations systematically, deduplicate aggressively, and maintain explicit measured mixing ratios between data categories.

Applied to CKE: Treat the scene bundle corpus like Nemotron treats code data. Generate synthetic variations systematically. Quality-score every row before training. Maintain explicit mixing ratios with named buckets. Spec18 started doing this with named dataset buckets — keep going.

NVIDIA Nemotron-v3 Pre-training Collection • Nemotron Nano 2 Tech Report

Where To Store The Evidence

Training conclusions are only useful if they remain attached to the exact run that produced them. That means audits, probe reports, failure autopsies, compiler smoke previews, dataset manifests, and run briefs should live in the cache-root run artifact folder under ~/.cache/ck-engine-v7/models/train/<run>/, not only in repo-side scratch notes.

Applied to CKE: Treat each run directory as the canonical retrieval unit for LFS export, IR Hub browsing, and later operator review. The repository should keep scripts, templates, and documentation; the run directory should keep the generated evidence.

Deduplication: Outsized Impact on Generalization

Deduplication of training data has outsized impact on generalization because it removes the memorization shortcut. When the same (or near-same) row appears many times, the model learns to replay it rather than learning the underlying structure.

Applied to CKE: Spec14a’s block-ordered corpus with 48 identical rows in sequence is the worst case for this. Interleave, deduplicate, and diversify.

Lee et al., 2022 — Deduplication of Training Data Makes Language Models Better

Memorization Without Overfitting

Some memorization is actually necessary for tail distribution generalization. The model needs to memorize rare patterns to generalize from them. The problem is when memorization becomes the only strategy because the data doesn’t create enough generalization pressure.

Applied to CKE: Spec14a’s default-board collapse is the textbook case: the model memorized the head of the distribution and ignored the tail. Weight decay + longer training + diverse data creates the pressure to move past memorization.

Feldman & Zhang, 2020 — What Neural Networks Memorize and Why

ELECTRA: Efficient Structural Pre-Training

Replaced token detection is a more efficient pre-training signal than masked language modeling, and it is closer to a structural correctness signal than cross-entropy loss. For a formal DSL, a “detect the corrupted scene field” objective could create stronger structural priors.

Applied to CKE: Instead of pure next-token prediction, consider a discriminative objective: present the model with a scene bundle where one field is wrong (wrong family, wrong form, wrong tone) and train it to identify the corrupted field.

Clark et al., 2020 — ELECTRA: Pre-training Text Encoders as Discriminators

How to Think About Training From Now On

The Fundamental Shift: From Artisanal to Systematic

The project has been treating training like craftsmanship — hand-map assets, hand-tune repair rows, hand-balance family pressure, hand-read autopsies. This doesn’t scale and creates the rung-repair trap. The shift should be toward systematic, compiler-driven data generation.

1. Compiler-As-Oracle: Generate Training Data Programmatically

You have a deterministic compiler. This is your biggest asset. Use it backward:

For each (family, form) in family_registry: For each (theme, tone, density) in style_combinations: For each topology_config in valid_topologies(family, form): bundle = generate_scene_bundle(family, form, theme, tone, density, topology) svg = compile(bundle, content_template) if svg validates: add_to_corpus(prompt_template(family, form, ...), bundle)

This gives you thousands of structurally perfect training rows with controlled variation along every axis. No hand-mapping. Closure/stop hygiene rows are still legitimate as a tightly capped fraction (5-10%), but the corpus backbone should be compiler-verified clean rows, not repair-heavy curricula.

2. Fix the Contract, Then Train Long (Grokking Approach)

Stop changing the contract after every few rungs. Instead:

  • Fix the scene bundle grammar once
  • Fix the tokenizer once
  • Fix the prompt surface once
  • Generate a much larger, clean, programmatic corpus via compiler-generated lattice
  • Then and only then: train for much longer with weight decay on that frozen recipe
  • Watch for the grokking transition: memorization → sudden generalization

The current pattern of 4-12 rungs per spec with curriculum tweaks between each is preventing grokking by constantly resetting the learning surface. But “train longer” is only a good experiment after steps 1-4 above are done. Training longer on a small noisy corpus is not the answer.

3. Separate Routing from Scene Planning

Spec17 and spec18 proved that intent → family+form+style+topology is genuinely harder than explicit controls → scene bundle. The right decomposition:

  • Router (separate model or deterministic rules): topic + goal + audience → family + form + style_defaults
  • Scene model: explicit family + form + style → scene bundle

The scene model is already proven to work under explicit control (spec16 R9). The routing can be a separate, potentially simpler model or even rule-based. This is how AlphaCode works — decompose the hard problem into an easier planning step and a constrained generation step.

4. Use pass@k + Compiler Filtering as Diagnostic and Inference Tactic

Do not require greedy exact match as the only way to understand model capability. Use pass@k as a side analysis:

candidates = [model.generate(prompt, temperature=0.3) for _ in range(10)] valid = [c for c in candidates if compiler.validates(c)] best = rank_by_structural_score(valid)

This leverages the compiler as a free structural verifier at inference time. However, promotion should still be based on greedy held-out exactness. pass@k is diagnostic (“how close is the model?”) and practical (“can we get usable output?”), but it does not mean training is solved.

5. Textbook-Style Data (phi-1 Approach)

For the routing problem specifically, generate “textbook” training data with explicit pedagogical structure:

# Lesson 1: Memory maps have regions and spans [intent: show memory layout] → family: memory_map [intent: compare memory sizes] → family: memory_map # Lesson 2: Timelines have sequential stages [intent: show project phases] → family: timeline [intent: compare before/after] → family: timeline # Lesson 3: Distinguish similar intents [intent: show component connections] → family: system_diagram (NOT memory_map) [intent: show data flow between layers] → family: system_diagram (NOT timeline)

Each lesson teaches one distinction with explicit contrast. This is what spec18 contrast scaffolds attempted, but mixed into noisy curricula. Separate them into a clean pedagogical progression.

6. Deterministic Pre-Checks Before Every Training Dollar

Make these automated gates, not manual reviews:

BEFORE TRAINING: ✓ Blueprint matches generated catalog (spec17 R2 bug) ✓ Token budget covers ≥1 full epoch/stage (spec17 R2 budget bug) ✓ No row appears >3x in sequence (spec14a ordering bug) ✓ Seed checkpoint probes non-zero on anchors (seed integrity gate) ✓ Compiler round-trips all gold scenes (compiler-first rule) ✓ Mixing ratios match bucket manifest (spec18 bucket discipline) ✓ Tokenizer covers all gold tokens, no <unk> (tokenizer gate) AFTER TRAINING: ✓ Held-out exact > 0 (hard gate) ✓ No family regression > 10% from baseline (non-regression gate) ✓ Raw AND repaired metrics reported separately (honest accounting)

7. Accumulate Rather Than Reset

Instead of starting each spec from scratch, consider curriculum continuation:

  • Phase 1: Train on explicit-control bundles until 100% exact (proven to work)
  • Phase 2: Gradually introduce intent prompts while keeping 50%+ explicit anchors
  • Phase 3: Gradually reduce explicit anchors as intent routing improves

This mirrors how distillation works — start from a strong known capability and gradually shift the distribution, rather than jumping to the hard task cold.

Rungs As Coverage Sensors

Once a branch becomes stable enough, a rung should teach us which region of the task is under-covered, not which exact prompts to memorize. This is the operational difference between healthy curriculum iteration and the old narrow-fix treadmill.

The Healthy Rung Loop

1. Train a stable rung on a frozen contract. 2. Autopsy misses by class: family, form, style, syntax, topology. 3. Translate those misses into generalized curriculum gaps. 4. Add broader neighbor coverage plus replay of the winning corpus. 5. Re-run and check whether the miss class shrinks without broad regression.

The key rule: do not answer a miss by teaching the exact held-out prompt. Answer it by teaching the surrounding region of the task so the next rung gets a fuller flavor of the data distribution.

How To Read Progress Correctly

If a new rung improves several regions while regressions stay limited and structured, that is a learnability signal. It means the branch is no longer mostly collapsing; it is exposing deterministic curriculum gaps that can be engineered.

  • Bad signal: broad collapse, reset-to-zero behavior, or diffuse regressions across all splits.
  • Good signal: one error class disappears, another shrinks, and the remaining misses cluster into a few interpretable classes.
  • Interpretation: the curriculum mix is still incomplete, not the whole branch concept.
Applied to spec19: the line is now stable enough that rung-to-rung miss-shift is meaningful. The work is no longer guessing or resetting; it is coverage engineering. That means spec19 stays in rung mode, not new-spec mode, until the remaining miss classes stop shrinking.

The Working Thesis for Next Training

Post-review consensus (2026-04-02): The analysis above is directionally right. The compiler-as-data-factory point is the strongest part. The “fix once, train long” point is also right, but only after the dataset becomes much larger and cleaner. The routing/planning split is a serious practical option, not just a fallback.

If we continue to the next training run, this is the working thesis:

StepActionWhy
1 Freeze the current bundle contract and tokenizer. No more simultaneous contract changes. One stable learning surface.
2 Generate a much larger compiler-verified lattice of clean rows. Use the compiler backward to systematically cover the family×form×style×topology combinatorial space. This is the phi-1 insight applied to CKE: dense, structured, pedagogically ordered data beats hand-mapped artisanal rows.
3 Run one long same-size experiment on that frozen recipe. Test whether the generalization threshold can be crossed at current capacity with enough clean data and training time. This is the Grokking experiment: stable contract, large clean corpus, weight decay, extended training.
4 In parallel or immediately after, run a same-data capacity canary. Cleanly separate the capacity question from the data question. If step 3 still shows zero held-out generalization on a large clean corpus, capacity may actually be the bottleneck for the intent-routing task specifically. This is the experiment that would prove it.
5 Keep pass@k as side analysis, not the main success criterion. Promotion should still be based on greedy held-out exactness. pass@k is diagnostic and practical but does not prove training quality.

This sequence is ordered by information value. Steps 1-2 are prerequisites. Step 3 tests the main hypothesis (data quality + training duration). Step 4 tests the alternative hypothesis (capacity). Step 5 keeps measurement honest throughout.

What This Thesis Does Not Claim

  • “Capacity is irrelevant” — it may be the bottleneck for intent routing. Step 4 tests this.
  • “Training longer always works” — it only makes sense after the corpus is large and clean. Step 2 is a prerequisite for step 3.
  • “Repair rows are bad” — closure/stop hygiene rows are still legitimate when tightly capped. The problem is repair-heavy curricula.
  • “The model just picks the wrong family” — on held-out rows, the failure is often upstream: the model is not emitting a parseable bundle at all. This is a generalization threshold problem, not just a routing problem.

The Mental Model: Tokenization → Output

When you look at a new training task, trace this chain mentally before spending any compute:

1. TOKENIZER: Can every target token be represented atomically? If not → fix tokenizer first (spec03 lesson) 2. CONTEXT BUDGET: Does the longest gold output fit in <80% of context? If not → compress DSL or raise context (spec12 token budget lesson) 3. COMPILER ROUND-TRIP: Can the compiler reproduce every gold asset from the DSL? If not → fix compiler first (the compiler-first principle) 4. DATA QUALITY: Is every training row compiler-verified and structurally valid? If not → generate programmatically, don’t hand-map (phi-1 lesson) 5. DATA DIVERSITY: Does the corpus cover all axis combinations evenly? If not → generate systematically across the combinatorial space 6. MEASUREMENT: Are probes testing the right thing? Is infrastructure correct? If not → fix probes first (spec17 R2 catalog bug lesson) 7. TRAINING BUDGET: Is there enough compute for ≥5 full epochs with weight decay? If not → train longer before changing anything (Grokking lesson) 8. ONLY THEN: If all above are clean and the model still plateaus → consider capacity increase

This ordering is not arbitrary. It reflects the actual failure history of the project: most wasted rungs were caused by problems at levels 1-6, not level 8.

The Superpower We’re Underusing

The Deterministic Compiler Is the Single Biggest Advantage This Project Has

Most ML projects would kill for a structural oracle. Stop treating it as a downstream consumer and start treating it as:

Data Factory Generate unlimited clean training rows
Diagnostic Gate pass@k filtering at inference; greedy exactness for promotion
Measurement Oracle Automated pre/post-training verification

TL;DR — Why we’re struggling: We keep changing the training contract too often, using hand-crafted repair-heavy curricula that create brittleness, stopping before grokking can occur, and trying to make one tiny model do both routing and scene planning. On held-out rows, the model often cannot even emit a parseable bundle — a generalization threshold problem upstream of family misrouting. Each narrow rung repair fixes one thing and breaks another, creating a treadmill.
How to break out: Generate training data programmatically from the compiler (phi-1 approach). Fix the contract once and train much longer on the larger clean corpus (Grokking approach). Separate routing from scene planning (AlphaCode approach). Use pass@k + compiler filtering as diagnostic and inference tactic, but keep greedy held-out exactness as the promotion gate (Codex approach). Automate all measurement gates so infrastructure bugs never waste another rung. Test capacity as a clean separate experiment, not as a guess.
