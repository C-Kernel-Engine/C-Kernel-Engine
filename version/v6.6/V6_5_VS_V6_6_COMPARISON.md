# V6.5 vs V6.6 Memory Layout Comparison

This document analyzes the differences between v6.5 (working) and v6.6 (broken) memory planning, and provides a detailed action plan to fix v6.6.

---

## Executive Summary

| Aspect | V6.5 (build_ir_v3.py) | V6.6 (build_ir_v6_6.py) | Issue |
|--------|----------------------|------------------------|-------|
| **Allocation Model** | Single contiguous `base` pointer | 6 separate `aligned_alloc` calls | Fragmented memory |
| **Buffer Definition** | Explicit per-layer buffers | Shared scratch buffers | Buffer aliasing bugs |
| **Offset Source** | BumpAllocator computes ALL offsets | Weights from manifest, activations computed | Mismatch in codegen |
| **Header Generation** | Full C header with structs | JSON intermediate, codegen embeds | Codegen ignores JSON offsets |
| **Weight + Activation** | Same contiguous region | Separate regions | Extra pointer arithmetic |
| **Canaries** | Yes, per-buffer boundary | No | No corruption detection |

---

## Detailed Analysis

### 1. Memory Allocation Model

#### V6.5: Single Contiguous Allocation

```python
# build_ir_v3.py line 605-638
class BumpAllocator:
    def __init__(self, start_offset: int = 0, alignment: int = CACHE_LINE):
        self.offset = start_offset  # Single counter

    def alloc(self, name: str, size: int) -> int:
        self.offset = align_up(self.offset, self.alignment)
        offset = self.offset
        self.offset += size  # Just increment!
        return offset
```

**Generated Model Struct (v6.5):**
```c
typedef struct {
    void *base;         // ONE pointer to everything
    size_t total_size;
} QWEN2_DECODEModel;

// Access anything via offset from base
#define QWEN2_DECODE_PTR(model, offset) \
    ((float*)((char*)(model)->base + (offset)))
```

#### V6.6: Multiple Separate Allocations

```c
// Generated by codegen_v6_6.py
typedef struct {
    uint8_t* bump_weights;    // Allocation 1
    float* activations;       // Allocation 2
    float* kv_cache;          // Allocation 3
    float* rope_cos;          // Allocation 4
    float* rope_sin;          // Allocation 5
    float* logits;            // Allocation 6
} CKModel;
```

**Problem:** Multiple allocations require:
- 6 separate `aligned_alloc()` calls
- 6 separate error checking paths
- Complex pointer management in codegen
- No unified offset scheme

### 2. Buffer Definition Strategy

#### V6.5: Explicit Per-Layer Buffers

Every layer gets its own activation buffers with unique offsets:

```python
# build_ir_v3.py line 644-727
def build_layer_layout(allocator, config, layer_id):
    buffers = []

    # Each layer has dedicated buffers
    buffers.append(allocator.alloc_buffer(f"layer.{layer_id}.ln1_gamma", [E], dtype, "weight"))
    buffers.append(allocator.alloc_buffer(f"layer.{layer_id}.ln1_out", [T, E], dtype, "activation"))
    buffers.append(allocator.alloc_buffer(f"layer.{layer_id}.wq", [H*D, E], dtype, "weight"))
    buffers.append(allocator.alloc_buffer(f"layer.{layer_id}.q", [T, H, D], dtype, "activation"))
    buffers.append(allocator.alloc_buffer(f"layer.{layer_id}.k", [T, KV, D], dtype, "activation"))
    buffers.append(allocator.alloc_buffer(f"layer.{layer_id}.v", [T, KV, D], dtype, "activation"))
    buffers.append(allocator.alloc_buffer(f"layer.{layer_id}.scores", [H, T, T], dtype, "activation"))
    # ... 20+ buffers per layer
```

**Generated Header (v6.5):**
```c
typedef struct {
    size_t ln1_gamma;    // [896]
    size_t ln1_out;      // [32768, 896]
    size_t wq;           // [14*64, 896]
    size_t bq;           // [14*64]
    size_t q;            // [32768, 14, 64]
    size_t k;            // [32768, 2, 64]
    size_t v;            // [32768, 2, 64]
    size_t scores;       // [14, 32768, 32768]  <- MASSIVE!
    // ...
} QWEN2_DECODELayerOffsets;

static const QWEN2_DECODELayerOffsets QWEN2_DECODE_LAYERS[24] = {
    [0] = { .ln1_gamma = 0x0FDB6400, .ln1_out = 0x0FDB7240, ... },
    [1] = { .ln1_gamma = 0xEC3817DC0, ... },
    // ...
};
```

#### V6.6: Shared Scratch Buffers

V6.6 reuses buffers across layers:

```python
# build_ir_v6_6.py line 1212-1227
# Only these buffers exist (shared by ALL layers):
add_buffer("q_scratch", q_size, f"[{num_heads}, {seq_len}, {head_dim}]")
add_buffer("attn_scratch", attn_out_size, f"[{num_heads}, {seq_len}, {head_dim}]")
add_buffer("mlp_scratch", mlp_size, f"[{seq_len}, {intermediate_size * 2}]")
add_buffer("layer_output", layer_out_size, f"[{seq_len}, {embed_dim}]")
```

**Problem:**
- Buffer aliasing: same buffer used for input AND output
- No residual storage: residual connections overwrite data
- Ping-pong broken: `layer_input` doesn't exist, only `layer_output`

### 3. Weight Offset Computation

#### V6.5: BumpAllocator Computes Offsets

V6.5 computes weight offsets fresh using `BumpAllocator`:

```python
# Weights are allocated in sequence
header_buffers.append(allocator.alloc_buffer("token_emb", [V, E], dtype, "weight"))
# ... allocator.offset is now token_emb_size

for layer_id in range(num_layers):
    layer.append(allocator.alloc_buffer(f"layer.{layer_id}.wq", [H*D, E], dtype, "weight"))
    # ... allocator.offset keeps incrementing
```

**Key:** Weights are sized based on HuggingFace config dtype (bf16), not GGUF quant dtype.

#### V6.6: Offsets from GGUF Manifest

V6.6 reads offsets directly from the manifest (which came from GGUF):

```python
# build_ir_v6_6.py line 935-949
for entry in entries:
    name = entry["name"]
    size = entry.get("size", 0)  # Size from GGUF converter

    all_weights[name] = {
        "bump_offset": current_offset,  # Just accumulate
    }
    current_offset += size
```

**This IS correct** - using actual GGUF sizes. But then...

### 4. Codegen Offset Usage

#### V6.5: Direct Struct Access

```c
// Generated code (v6.5)
void decode_layer(QWEN2_DECODEModel *model, int layer) {
    const QWEN2_DECODELayerOffsets *L = &QWEN2_DECODE_LAYERS[layer];

    // Clean: offset is a single lookup
    rmsnorm_forward(
        QWEN2_DECODE_PTR(model, L->ln1_out),    // Exact offset
        QWEN2_DECODE_PTR(model, L->ln1_gamma),  // Exact offset
        ...
    );

    // Q projection
    gemm_nt_q5_0(
        QWEN2_DECODE_PTR(model, L->ln1_out),    // Input
        QWEN2_DECODE_PTR(model, L->wq),         // Weight
        QWEN2_DECODE_PTR(model, L->q),          // Output
        ...
    );
}
```

#### V6.6: Ignores Computed Offsets!

The codegen (`codegen_v6_6.py`) **does NOT use the offsets from lowered IR**:

```python
# codegen_v6_6.py line 510-525
elif source_type == "activation":
    offset = int(source_ref) if source_ref else 0
    # ↑ Uses raw int from binding, NOT from IR lower 2!
```

**Generated code (v6.6):**
```c
// BROKEN: Everything uses offset 0!
gemv_q5_0_q8_0(
    ((float*)(model->activations + 0)),  // Always 0!
    W_PTR(L->wq),
    ((void*)(model->activations + 0)),   // Always 0!
    EMBED_DIM, EMBED_DIM
);
```

**Bug:** The lowered IR has correct offsets (`activation_offset: 931443712`) but codegen ignores them!

### 5. Per-Layer Dtype Arrays

#### V6.5: Complete Dtype Tracking

```c
// v6.5 header tracks dtype per layer per weight
static const CKDataType QWEN2_DECODE_LAYER_WV_DTYPE[] = {
    CK_DT_Q8_0, CK_DT_Q8_0, CK_DT_Q5_0, CK_DT_Q8_0, ...
};

static const CKDataType QWEN2_DECODE_LAYER_W2_DTYPE[] = {
    CK_DT_Q6_K, CK_DT_Q6_K, CK_DT_Q4_K, CK_DT_Q6_K, ...
};
```

This enables:
```c
// Select kernel based on layer's dtype
if (QWEN2_DECODE_LAYER_W2_DTYPE[layer] == CK_DT_Q4_K) {
    gemv_q4_k_q8_k(out, W_PTR(L->w2), in, M, K);
} else {
    gemv_q6_k_q8_k(out, W_PTR(L->w2), in, M, K);
}
```

#### V6.6: Dtypes in IR but Not Header

V6.6 has dtype info in the lowered IR JSON:
```json
"weights": {
    "w2": { "dtype": "q4_k", "bump_offset": 303089240 }
}
```

But no C header with dtype arrays is generated. Codegen hardcodes kernel names.

### 6. What V6.6 Kernel Maps Need

The kernel maps in `version/v6.6/kernel_maps/` are metadata files. For header generation, we need:

| Required Field | Current Status | Action |
|----------------|----------------|--------|
| `inputs[].shape` | Has symbolic shapes like `["M", "K"]` | Need to resolve to concrete values |
| `outputs[].shape` | Has symbolic shapes | Need to resolve |
| `impl.c_declaration` | Present | Good |
| `impl.function` | Present | Good |
| Weight dtype binding | Implicit via kernel name (e.g., `gemv_q5_0_q8_0`) | Needs explicit mapping |

**Example: gemv_q5_0_q8_0.json**
```json
{
  "id": "gemv_q5_0_q8_0",
  "op": "gemv",
  "inputs": [
    {"name": "W", "dtype": "q5_0", "shape": ["M", "K"]},
    {"name": "x", "dtype": "q8_0", "shape": ["K"]}
  ],
  "outputs": [
    {"name": "y", "dtype": "fp32", "shape": ["M"]}
  ]
}
```

**Missing:** No "buffer_requirements" or "scratch_size" field.

| `template` | Loaded from templates/ | Present | OK |

## Action Plan

### Phase 1: Fix Memory Layout Generator

**File:** `version/v6.6/scripts/build_ir_v6_6.py`

1. **Add Per-Layer Activation Buffers**
   - Replace shared `q_scratch`, `attn_scratch`, `mlp_scratch` with per-layer buffers
   - OR implement proper buffer reuse with validation

2. **Add Double-Buffering for Residuals**
   ```python
   add_buffer("residual_a", layer_size, "[seq_len, embed_dim]")
   add_buffer("residual_b", layer_size, "[seq_len, embed_dim]")
   ```

3. **Compute Total Memory Correctly**
   ```python
   total_size = weights_size + activations_size + kv_cache_size + rope_size + logits_size
   ```

### Phase 2: Create Header Generator

**New File:** `version/v6.6/scripts/gen_layout_header_v6_6.py`

Generate a C header like v6.5:

```python
def emit_header(layout: Dict, output_path: str):
    """Generate C header with:
    - Model config #defines
    - HeaderOffsets struct
    - LayerOffsets struct
    - Per-layer dtype arrays
    - FooterOffsets struct
    - Accessor macros
    """
```

**Output:** `{model}_layout.h`

### Phase 3: Fix Codegen

**File:** `version/v6.6/scripts/codegen_v6_6.py`

1. **Include Generated Header**
   ```c
   #include "qwen2_layout.h"
   ```

2. **Use Struct Offsets**
   ```c
   // Replace:
   ((float*)(model->activations + 0))

   // With:
   QWEN2_DECODE_PTR(model, QWEN2_DECODE_ACT.buffer_a)
   ```

3. **Remove Multiple Allocations**
   ```c
   // Replace:
   model->bump_weights = aligned_alloc(64, WEIGHTS_SIZE);
   model->activations = aligned_alloc(64, ACT_SIZE);
   // ...

   // With:
   model->base = aligned_alloc(64, QWEN2_DECODE_TOTAL_BYTES);
   ```

### Phase 4: Update Kernel Maps

For each kernel map, add buffer requirements:

```json
{
  "id": "gemv_q5_0_q8_0",
  "buffer_requirements": {
    "input": {"min_size": "K * sizeof(q8_0_block)", "alignment": 64},
    "output": {"min_size": "M * 4", "alignment": 64},
    "scratch": {"size": 0}
  }
}
```

### Phase 5: Validation

1. **Offset Bounds Checking**
   ```python
   def validate_offsets(layout):
       for op in layout["operations"]:
           for buf in op["activations"].values():
               assert buf["offset"] < layout["memory"]["activations"]["size"]
   ```

2. **Canary Generation**
   - Add canary markers at buffer boundaries
   - Verify canaries at runtime

3. **Numerical Parity Test**
   - Run v6.5 and v6.6 on same input
   - Compare logits output
