Why Greedy Longest-Match Fails

Hash table and Trie implementations use greedy longest-match, which doesn't achieve 100% parity with BPE tokenizers:

The Problem

Text: "lower"

Greedy Longest-Match:
  → Found "lower" in vocab → [12345]  (1 token)

True BPE (with merge rules):
  → Initial: ['l', 'o', 'w', 'e', 'r']
  → Apply merges in priority order:
    • Merge 'e' + 'r' → 'er' (priority 50)
    • Merge 'o' + 'w' → 'ow' (priority 100)
    • Merge 'l' + 'ow' → 'low' (priority 200)
    • Merge 'low' + 'er' → 'lower' (priority 500)
  → Final: [12345]

// But for "lowest":
Greedy: "low" + "est" → different tokens than HF
BPE:    Apply merges → matches HF exactly

The difference: greedy always picks the longest token, but BPE applies merges in priority order, which can produce different tokenizations.

True BPE: 100% HuggingFace Parity

The True BPE implementation achieves exact token-for-token parity with HuggingFace by implementing the actual BPE algorithm:

The Algorithm

  1. Pretokenize: Split text using GPT-2 regex patterns
  2. Byte-Level Encode: Convert bytes to GPT-2 characters (space → Ġ)
  3. Initialize: Split into individual characters
  4. Merge Loop: Find and apply highest-priority merge
  5. Lookup: Convert final tokens to IDs

GPT-2 Pretokenizer

HuggingFace tokenizers use a regex-based pretokenizer that splits text BEFORE applying BPE. This is crucial for parity:

// GPT-2 pretokenizer patterns (from HuggingFace):
Pattern 1:  \'s|\'t|\'re|\'ve|\'m|\'ll|\'d     // Contractions
Pattern 2:  [^\r\n\p{L}\p{N}]?\p{L}+          // Words with optional prefix
Pattern 3:  \p{N}{1,3}                        // Numbers (1-3 digits)
Pattern 4:   ?[^\s\p{L}\p{N}]+[\r\n]*         // Punctuation with space
Pattern 5:  \s*[\r\n]+                        // Newlines
Pattern 6:  \s+(?!\S)                         // Trailing spaces
Pattern 7:  \s+                               // Other whitespace

// Example: "def fibonacci(n):"
Pretokenized: ["def", " fib", "onacci", "(", "n", "):"]
               ↑       ↑                 ↑
             word   space+word      punctuation

Hash Table vs Trie: The Performance Journey

Our tokenizer went through a major optimization from hash table to trie-based lookups:

Why the Hash Table Was Slow

The original hash table approach had O(n × k) complexity for longest-match tokenization:

// For each position, try all possible lengths
for (size_t len = max_len; len >= 1; len--) {
    char tmp[65];
    memcpy(tmp, text + pos, len);    // Copy bytes
    tmp[len] = '\0';                  // Null terminate

    TokenInfo *info = hash_table_lookup(vocab, tmp);
    if (info) {
        best_id = info->id;           // Found!
        break;
    }
}

Why Trie is Fast

The trie approach has O(k) complexity for each position:

// Single traversal through trie
CKTrieNode *node = trie->root;
CKTrieNode *last_token = NULL;
size_t pos = start_pos;

while (pos < text_len && node->children[text[pos]]) {
    node = node->children[text[pos]];
    pos++;

    if (node->token_id >= 0) {
        last_token = node;           // Remember last valid token
        last_token_len = pos - start_pos;
    }
}

API Reference

True BPE API (Recommended for HuggingFace Parity)

// Create True BPE tokenizer
CKTrueBPE *bpe = ck_true_bpe_create();

// Add tokens from vocabulary
ck_true_bpe_add_token(bpe, "hello", 100, 0.0f);
ck_true_bpe_add_token(bpe, "Ġworld", 101, 0.0f);  // Ġ = space prefix

// Add merge rules (from tokenizer.json)
ck_true_bpe_add_merge_by_tokens(bpe, "he", "llo", 0);  // priority 0 = first
ck_true_bpe_add_merge_by_tokens(bpe, "hel", "lo", 1);  // priority 1 = second

// Set special tokens
ck_true_bpe_set_special_ids(bpe, unk_id, bos_id, eos_id, pad_id);

// Encode text
int32_t ids[1024];
int num_ids = ck_true_bpe_encode(bpe, text, strlen(text), ids, 1024);

// Cleanup
ck_true_bpe_free(bpe);

Greedy Matcher API (Faster, ~85% Parity)

// Create BPE tokenizer
CKTokenizer *tok = ck_tokenizer_create(CK_TOKENIZER_BPE);

// Create WordPiece tokenizer
CKTokenizer *tok = ck_tokenizer_create(CK_TOKENIZER_WORDPIECE);

// Create SentencePiece tokenizer
CKTokenizer *tok = ck_tokenizer_create(CK_TOKENIZER_SPM);

Adding Tokens

// Add regular token
ck_tokenizer_add_token(tok, "hello", 100, 0.0f);

// Add special token
ck_tokenizer_add_special_token(tok, "<unk>", 0);
ck_tokenizer_add_special_token(tok, "<s>", 1);
ck_tokenizer_add_special_token(tok, "</s>", 2);

Encoding Text

// Encode text to token IDs
int32_t ids[100];
int num_ids = ck_tokenizer_encode(tok, text, -1, ids, 100);

// num_ids contains the number of tokens written
// ids[0..num_ids-1] contains the token IDs

Switching Lookup Methods

// Enable trie-based lookups (DEFAULT - faster)
ck_tokenizer_set_use_trie(tok, true);

// Use hash table lookups (slower, for comparison)
ck_tokenizer_set_use_trie(tok, false);
