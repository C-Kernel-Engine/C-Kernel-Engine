### What is llama.cpp doing that we're not?

1. **GEMM kernel efficiency** - Better SSE blocking, prefetching, loop unrolling
2. **Memory layout** - Optimized for cache (weights packed, activations contiguous)
3. **No malloc in hot path** - Arena allocation or pre-allocated buffers
4. **Better SIMD utilization** - More aggressive SSE usage, less scalar code
5. **Reduced memory copies** - Direct pointers, no intermediate buffers
6. **Quantization strategy** - More efficient Q4_K dequantization
7. **KV cache optimization** - Better cache locality

### Key Finding: Activations are FP32

Looking at `gemm_kernels_q4k.c`:
```c
const float *xp = &x[b * QK_K + iter * 64];  // x is FP32!
```

The kernel:
1. Takes Q4_K weights (4-bit quantized, 0.5 bytes/weight)
2. Takes **FP32 activations** (4 bytes/element)
3. Dequantizes weights to FP32 on-the-fly
4. Does FP32 matrix multiplication
5. Outputs FP32

### W8A16 (INT8 Activations) Analysis - NOT RECOMMENDED FOR YOUR CPU

**What W8A16 would be:**
- **W**: Weights = Q4_K (4-bit) or could be INT8
- **8**: Activations = INT8 (1 byte/element)
- **16**: Accumulation = INT16 (or use VNNI for INT32)

**BUT W8A16 won't help on your CPU!**
- No VNNI = INT8 ops use SSE, not faster than FP32
- 4x memory bandwidth reduction is offset by:
  - No hardware INT8 multiply-accumulate
  - Extra quantization/dequantization overhead
  - Still falls back to 128-bit SSE

**The real bottlenecks on YOUR CPU:**
1. **GEMM kernel efficiency** - SSE only (128-bit vs 256-bit AVX2)
2. **Memory allocation** - per-batch malloc overhead
3. **Scalar operations** - RMSNorm, Softmax not vectorized
4. **No FMA** - Can't do multiply-add in one cycle

**Recommendation for your CPU:**
- Focus on SSE optimizations (already present)
- Memory allocation pool
- Better cache blocking for SSE
- Precompute tables (RoPE cos/sin)

### The Problem: Memory-Bound Attention

Current `attention_decode_fused.c` already fuses 10 kernels into 1, but intermediates still flow through stack/heap:

```
Current (good, but not mega-fused):
  RMSNorm → [ln1_row on stack: 8KB] → QKV → [q/k/v_token on heap: 20KB]
                                            ↓
                                    RoPE → [in-place]
                                            ↓
                                    Flash Attn → [attn_token on heap: 8KB]
                                            ↓
                                    OutProj → [DRAM]

Result: ~32KB intermediates per layer × 24 layers = 768KB per token!
```

### The Solution: Mega-Fusion

```
Mega-Fused (holy grail):
  RMSNorm → [registers] → QKV → [L2]
                      ↓          ↓
               [registers] ← RoPE ←┘
                      ↓
  Flash Attn → [registers] → OutProj
                      ↓
          Residual Add (in store)
                      ↓
           [DRAM: 4KB input + 4KB output only]

Result: ~8KB total per layer × 24 layers = 192KB per token (4× reduction)
```

### Memory Traffic Comparison

| Metric | Current v6.5 | Mega-Fused | Reduction |
|--------|--------------|------------|-----------|
| DRAM Reads | ~40KB × 24 | 4KB | **240×** |
| DRAM Writes | ~40KB × 24 | 4KB | **240×** |
| Arithmetic Intensity | ~0.5 FLOPs/byte | ~4000 FLOPs/byte | **8000×** |
| Bottleneck | Memory | Compute | - |

### Expected Speedup: 5-10×

Moving from memory-bound to compute-bound is the single biggest optimization possible.

### The Problem: Current Kernels Write to DRAM

Existing kernels (`rmsnorm_kernels.c`, `gemm_kernels.c`, `attention_flash_true.c`) are designed to:
1. Read input from memory
2. Write output to memory
3. This causes ~800KB of intermediate traffic per layer!

**Current kernel design (not fusion-friendly):**
```c
// rmsnorm_kernels.c - writes to output array
void rmsnorm_forward(input, gamma, output, ...);

// gemm_kernels.c - writes to output array
void gemm_blocked_serial(input, weights, output, ...);

// attention_flash_true.c - writes to output array
void flash_attention_forward(Q, K, V, output, ...);
```

**Each call writes intermediates to stack/heap, which may spill to DRAM.**

### The Solution: Fused Kernels with Buffer Passing

New kernel API designed for fusion:
```c
// Fused kernels take output buffer as parameter
void rmsnorm_fused(input, gamma, output_buffer, ...);
void gemm_fused(input, weights, output_buffer, ...);
void flash_attention_fused(Q, K, V, output_buffer, ...);

// Kernels can chain without DRAM intermediates:
// rmsnorm_fused → gemm_fused → rope_fused → flash_fused → outproj_fused
```

### Two-Level Fusion Strategy (from `per_head_fusion_math.svg`)

```
Level 1: Per-Head Fusion (fits in L1 per head)
┌─────────────────────────────────────────────────────────┐
│ Q_h,K_h,V_h → RoPE → Flash Attention → O_h             │
│                                                             │
│ Working set: 16.8KB (d=64, T=64, FP16)                   │
│ Fits in L1 cache!                                         │
│ Parallel across cores!                                    │
└─────────────────────────────────────────────────────────┘
         ↓ L2 buffer (O_h per head)
         ↓ Barrier sync
Level 2: Cross-Head Coordination
┌─────────────────────────────────────────────────────────┐
│ RMSNorm → QKV GEMM → distribute to cores → barrier → OutProj │
└─────────────────────────────────────────────────────────┘
```

### Cache Constraint Formula (Deterministic)

For per-head flash attention to fit in L1:
```
T_kv ≤ (S_L1 × 0.8 - 2×d×B) / (2×d×B)
```

Where:
- `T_kv` = KV tile size for flash attention
- `S_L1` = L1 cache size (48KB typical)
- `d` = head dimension (64)
- `B` = bytes per element (2 for FP16, 4 for FP32)

**Example for d=64, FP16, 48KB L1:**
```
T_kv = (48KB×0.8 - 256B) / 256B = 150 → use T_kv=128
```

### What to Look For in Profiles

| Symptom | Likely Cause | Fix |
|---------|--------------|-----|
| 80%+ time in GEMM | Poor SSE implementation | Match llama.cpp's GEMM |
| High malloc overhead | malloc in hot path | Arena allocator |
| Branch mispredictions | Data-dependent loops | Loop unrolling |
| Cache misses | Poor data layout | Pack weights, contiguous activations |
| Slow dequant | Inefficient Q4_K unpack | Optimize unpack + multiply |
| Scalar code | No SSE used | Vectorize RMSNorm, Softmax |
