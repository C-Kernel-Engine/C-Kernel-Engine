// Check if model is ready
char ready_json[PATH_MAX];

    printf("Preparing model (first run)...\n");

## Part 2: INT8 Activation Strategy

### Kernel Selection Matrix

| Weight Type | Inference Activation | Training Activation |
|-------------|---------------------|---------------------|
| Q4_K        | Q8_K (fast)         | FP32 (precise)      |
| Q5_0        | Q8_0 (fast)         | FP32 (precise)      |
| Q8_0        | Q8_0 (fast)         | FP32 (precise)      |
| Q6_K        | FP32 (no INT8 kern) | FP32                |
| FP32        | FP32                | FP32                |
| BF16        | BF16                | BF16                |

### Reasoning

**Inference (INT8 default):**
- Memory bandwidth is the bottleneck
- Q8_K activations: ~1.14 bytes/element (vs 4 bytes for FP32)
- 3-4x speedup for GEMV operations
- Quality loss is negligible (< 0.1% perplexity)

**Training (FP32 default):**
- Gradient precision matters for convergence
- Small numerical errors accumulate over many steps
- Speed is less critical than correctness
- Can optionally enable INT8 for faster iteration during prototyping

### Implementation

#### Codegen Changes
```python
# In v6.5_codegen.py

def get_activation_dtype(weight_dtype: str, mode: str) -> str:
    """Determine activation dtype based on weight type and mode."""

    # Training/backprop: always use full precision
    if mode in ("training", "backward"):
        return "fp32"

    # Inference: use INT8 if kernel exists
    INT8_SUPPORTED = {"q4_k": "q8_k", "q5_0": "q8_0", "q8_0": "q8_0"}
    return INT8_SUPPORTED.get(weight_dtype, "fp32")
```

#### CLI Flags (optional override)
```
--activations=auto     # Default: INT8 for inference, FP32 for training
--activations=int8     # Force INT8 (faster, slightly less precise)
--activations=fp32     # Force FP32 (slower, more precise)
```

### Performance Expectations

For a Q4_K model like Qwen2-0.5B:
```
FP32 activations:  ~1 tok/s (current)
INT8 activations:  ~3-4 tok/s (target)
```

The 3-4x comes from:
- 4x less memory read (Q8_K vs FP32)
- Faster INT8 dot products (AVX2/VNNI)
