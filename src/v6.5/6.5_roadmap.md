### Code Review of src/v6/ck_cli_v6.c

#### Positive Changes
- Threading model with pthread for separation of UI/engine
- Condition variables for synchronization between threads
- Working tokenizer integration via `ck_tokenizer_*` API

#### CRITICAL REGRESSION: Stop Token Detection
```c
// v6.5 code (BROKEN - hardcoded for Qwen only)

```

This regression breaks support for:
- **Llama2/3**: EOS is token 2 or 128009
- **Mistral**: EOS is token 2

- **Phi3**: EOS is token 3

The previous v6 implementation used an O(1) lookup table from `chat_template.json`:
```c
// v6 code (CORRECT - model-agnostic)
if (is_stop_token_id(next_token)) break;
```

#### Missing Features
- No cache folder scanning
- No model discovery/interactive selection
- No offset validation for bump files
- No parity testing infrastructure

## 3. Performance Optimization (30x Gap to llama.cpp)

### CPU Capability Analysis - BOTH RUN ON SAME CPU!

**Your CPU: Intel Core i7-3630QM (Ivy Bridge, 2012)**

| Feature | Available | Both C-Kernel & llama.cpp Use |
|---------|-----------|-------------------------------|
| AVX | Yes | Yes (256-bit float) |
| AVX2 | **NO** | Both use SSE fallback |
| FMA | **NO** | Both use separate mul+add |
| VNNI | **NO** | Both can't use it |
| SSE4.1/4.2 | Yes | Yes (128-bit ops) |

**CRITICAL INSIGHT:**
```
llama.cpp on i7-3630QM: ~30 tok/s  ✓
C-Kernel-Engine on i7-3630QM: ~1 tok/s  ✗

SAME CPU. DIFFERENT CODE.
```

The 30x gap is **NOT hardware** - it's **software implementation**.

This is GOOD NEWS - we can catch up by fixing our code!

### What is llama.cpp doing that we're not?

1. **GEMM kernel efficiency** - Better SSE blocking, prefetching, loop unrolling
2. **Memory layout** - Optimized for cache (weights packed, activations contiguous)
3. **No malloc in hot path** - Arena allocation or pre-allocated buffers
4. **Better SIMD utilization** - More aggressive SSE usage, less scalar code
5. **Reduced memory copies** - Direct pointers, no intermediate buffers
6. **Quantization strategy** - More efficient Q4_K dequantization
7. **KV cache optimization** - Better cache locality

### Current Precision Analysis (Jan 2025)

| Component | Current Precision | Bytes/Element | Notes |
|-----------|-------------------|---------------|-------|
| **Weights** | Q4_K (4-bit) | 0.5 | Quantized, dequantized on-the-fly |
| **Activations** | **FP32** | **4** | Full precision! |
| **GEMM Output** | FP32 | 4 | Final accumulate in float |
| **RMSNorm** | FP32 (has BF16/INT8 variants) | 4 | Default is FP32 |
| **Attention** | FP32 or BF16 | 4 or 2 | Has BF16 variant |
| **MLP (SwiGLU)** | FP32 or BF16 | 4 or 2 | Has BF16 variant |

### Key Finding: Activations are FP32

Looking at `gemm_kernels_q4k.c`:
```c
const float *xp = &x[b * QK_K + iter * 64];  // x is FP32!
```

The kernel:
1. Takes Q4_K weights (4-bit quantized, 0.5 bytes/weight)
2. Takes **FP32 activations** (4 bytes/element)
3. Dequantizes weights to FP32 on-the-fly
4. Does FP32 matrix multiplication
5. Outputs FP32

### W8A16 (INT8 Activations) Analysis - NOT RECOMMENDED FOR YOUR CPU

**What W8A16 would be:**
- **W**: Weights = Q4_K (4-bit) or could be INT8
- **8**: Activations = INT8 (1 byte/element)
- **16**: Accumulation = INT16 (or use VNNI for INT32)

**BUT W8A16 won't help on your CPU!**
- No VNNI = INT8 ops use SSE, not faster than FP32
- 4x memory bandwidth reduction is offset by:
  - No hardware INT8 multiply-accumulate
  - Extra quantization/dequantization overhead
  - Still falls back to 128-bit SSE

**The real bottlenecks on YOUR CPU:**
1. **GEMM kernel efficiency** - SSE only (128-bit vs 256-bit AVX2)
2. **Memory allocation** - per-batch malloc overhead
3. **Scalar operations** - RMSNorm, Softmax not vectorized
4. **No FMA** - Can't do multiply-add in one cycle

**Recommendation for your CPU:**
- Focus on SSE optimizations (already present)
- Memory allocation pool
- Better cache blocking for SSE
- Precompute tables (RoPE cos/sin)

### Current Performance
- C-Kernel-Engine: ~1 token/second
- llama.cpp: ~30 tokens/second (SAME CPU!)
- Gap: **30x**

### Bottleneck Analysis & Roadmap

We MUST profile to find WHERE we're slow. But based on typical patterns:

#### Phase 3a: Low-Hanging Fruit (Target: 5x improvement)
| Bottleneck | Current | Target | Fix |
|------------|---------|--------|-----|
| Memory allocation | Per-batch malloc | Arena/allocation pool |
| KV cache writes | memcpy per layer | Direct pointer arithmetic |
| Softmax | Scalar | SSE vectorization |
| Token sampling | Full vocab scan | Top-k heap |
| GEMM | Basic blocking | Better SSE blocking |

#### Phase 3b: Core Kernels (Target: 10x improvement)
| Kernel | Current | Target | Fix |
|--------|---------|--------|-----|
| GEMM (Q4_K) | Poor SSE | Match llama.cpp SSE |
| RMSNorm | Scalar | SSE vectorization |
| RoPE | Per-element | Precompute cos/sin tables |
| Attention | Basic | Cache-friendly layout |

#### Phase 3c: Advanced (Target: 20x improvement)
- [ ] Match llama.cpp's Q4_K efficiency
- [ ] KV cache optimization
- [ ] Batch decode (4-8 tokens parallel)
- [ ] Prefetch strategies

2. **Compare GEMM kernels** - llama.cpp's SSE impl vs ours

### W8A16 Implementation Plan

```

1. Quantize activations to INT8 before GEMM
   - Add quantization kernel: fp32 -> int8 with scale
   - Per-channel or per-tensor quantization

   - Apply scales and dequantize to FP32 output

3. Memory layout optimization
   - Use NHWC instead of NCHW for cache friendliness
```

```python
"""
Standardized benchmark for C-Kernel-Engine.

Usage:
    python benchmark.py --model Qwen2-0.5B --runs 3
    python benchmark.py --compare --baseline llama.cpp
"""

import subprocess
import time
import json
from pathlib import Path

def run_benchmark(model_path, runs=3):
    results = []
    for i in range(runs):
        start = time.time()
        # Run inference
        result = subprocess.run([
            "./build/ck-cli-v6",
            f"{model_path}/ck-kernel-inference.so",
            f"{model_path}/weights.bump",
            "--prompt", "Hello, I am a",
            "--max-tokens", "100",
            "--timing"
        ], capture_output=True, text=True)
        elapsed = time.time() - start
        results.append(elapsed)

    return {
        "model": str(model_path),
        "runs": runs,
        "mean_time": sum(results) / len(results),
        "min_time": min(results),
        "max_time": max(results),

    }
```
