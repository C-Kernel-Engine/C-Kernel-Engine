# C-Kernel-Engine v6 CLI Improvements

## Overview

This document tracks improvements to `ck_cli_v6` and the generated code.

---

## 1. CLI Features

### 1.1 Streaming Output

**Current behavior:** Buffers output, prints at end of generation.

**Desired behavior:** Stream tokens as they're generated for real-time feedback.

```c
// In output_token():
fflush(stdout);  // Flush after each token

// Optional: small delay for visual effect
usleep(10000);   // 10ms between tokens
```

**Implementation:**
```bash
ck-cli-v6 model.so weights.bump --stream
ck-cli-v6 model.so weights.bump -s          # short form
```

### 1.2 Timing & Tokens/sec

**Current behavior:** No timing information.

**Desired behavior:** Show detailed timing breakdown.

```c
// Track with clock_gettime(CLOCK_MONOTONIC)
struct timespec t_start, t_end;

// In run_prompt():
clock_gettime(CLOCK_MONOTONIC, &t_prefill_start);
api->forward(NULL);
clock_gettime(CLOCK_MONOTONIC, &t_prefill_end);
prefill_ms = (t_end - t_start) * 1000.0;

// In decode loop:
clock_gettime(CLOCK_MONOTONIC, &t_decode_start);
api->decode(next_token, NULL);
clock_gettime(CLOCK_MONOTONIC, &t_decode_end);
decode_ms += (t_end - t_start) * 1000.0;
decode_count++;
```

**Output format:**
```
ck-cli-v6 model.so weights.bump --timing

Assistant: Hello! How can I help you today?

Timing:
  Prefill:  21 tokens in 13939.30 ms (1.51 tok/s)
  Decode:    9 tokens in  6173.67 ms (1.46 tok/s) avg 685.96 ms/tok
  Sample:      12.14 ms / 10 runs
  Total:     30 tokens in 20126.69 ms
```

**Implementation:**
```bash
ck-cli-v6 model.so weights.bump --timing
ck-cli-v6 model.so weights.bump -T          # short form
```

### 1.3 Benchmark Mode

**Current behavior:** Shows output during generation.

**Desired behavior:** No output, just timing summary (for performance testing).

```bash
ck-cli-v6 model.so weights.bump --bench --warmup 5 --max-tokens 100

Benchmark (100 tokens, 5 warmup):
  Prefill:  21 tokens in 13939.30 ms
  Decode:   79 tokens in 54123.45 ms (684.60 ms/tok, 1.46 tok/s)
```

**Implementation:**
```bash
ck-cli-v6 model.so weights.bump --bench
ck-cli-v6 model.so weights.bump -b          # short form
```

### 1.4 Updated CLI Usage

```bash
ck-cli-v6 <lib.so> <weights.bump> [OPTIONS]

Options:
  --prompt, -p TEXT      Prompt (non-interactive mode)
  --max-tokens, -t N     Max tokens to generate (default: 128)
  --stream, -s           Stream tokens as generated
  --timing, -T           Show timing breakdown
  --bench, -b            Benchmark mode (no output)
  --warmup N             Warmup runs (default: 3)
  --context, -c N        Override context/KV cache size
  --eos IDS              Comma-separated EOS token IDs
  --ignore-eos           Do not stop on EOS tokens
  --help, -h             Show this help

Interactive Commands:
  /exit, /quit           Exit
  /reset                 Reset KV cache
  /help                  Show help
```

## 3. dlopen Impact Analysis

### 3.1 Does dlopen Slow Things Down?

**Short answer:** No significant impact for inference.

**Explanation:**

```c
// dlopen happens ONCE at startup
api->handle = dlopen(lib_path, RTLD_NOW);  // ~1-5ms

// Symbol resolution happens ONCE per function
dlsym(handle, "ck_model_forward");  // ~0.01ms each

// Actual inference: NO dlopen overhead
api->forward(NULL);  // Direct function pointer call
```

**Timing breakdown:**
```
Startup:
  dlopen()                              ~1-5ms
  dlsym() x 10 symbols                  ~0.1ms
  ck_model_init(weights.bump)           ~100-500ms (IO bound)
  ck_true_bpe_load_binary()             ~10-50ms

Per-generation:
  ck_model_embed_tokens()               ~1ms (tokenization)
  ck_model_forward()                    ~5000ms (COMPUTATION)
  ck_model_decode() x N                 ~N x 500ms (COMPUTATION)
  ck_model_sample_argmax()              ~0.1ms
```

**Conclusion:** dlopen is a one-time ~5ms cost at startup. For a 30-token generation taking 20 seconds, this is 0.025% overhead. Negligible.

### 3.2 Benefits of dlopen

1. **Model-specific binaries:** Each model has its own .so with explicit kernels
2. **No runtime code generation:** Code is compiled once, loaded dynamically
3. **Isolation:** Model code can't crash the CLI (in theory)
4. **Flexibility:** Different models can have different APIs

### 3.3 Alternative: Static Linking

For embedded systems where dlopen isn't available:

```c
// Instead of dlopen, link directly:
// gcc -o ck-cli-v6 ck_cli_v6.c model.c -lm -lpthread

// Tradeoff:
//   + No dlopen overhead (tiny)
//   - One binary per model (no shared CLI)
//   - Simpler deployment
```

## 4. Chat Templates for Better Responses

### 4.1 The Problem

Current behavior: Just predicts next token.

```python
# Python wrapper outputs:
# "As an artificial intelligence language model, I don't work directly."
```

This happens because:
1. No system prompt
2. No chat template
3. Model sees bare text, defaults to robotic response

### 4.2 Solution: Add --chat-template Flag

```bash
# Use HuggingFace-style chat template
ck-cli-v6 model.so weights.bump --chat-template qwen \
    --system "You are a helpful AI assistant."

# Or with prompt:
ck-cli-v6 model.so weights.bump --chat-template qwen \
    --system "You are a helpful AI assistant." \
    --prompt "Hello"
```

### 4.3 Chat Template Integration

**Qwen2 chat template:**
```
<|im_start|>system
You are a helpful AI assistant.<|im_end|>
<|im_start|>user
Hello!<|im_end|>
<|im_start|>assistant
```

**Implementation in C:**

```c
// Add to CLI options
typedef struct {
    ...
    bool use_chat_template;
    const char *chat_template;  // "qwen", "llama", "mistral"
    const char *system_prompt;
} CLIOptions;

// Apply template before tokenizing
char *formatted_prompt = NULL;
if (opt->use_chat_template) {
    formatted_prompt = apply_chat_template(
        opt->chat_template,
        opt->system_prompt,  // NULL if not provided
        user_prompt
    );
    input = formatted_prompt;
}
```

**Supported templates (in C):**
```c
// qwen
"<|im_start|>system\n%s<|im_end|>\n<|im_start|>user\n%s<|im_end|>\n<|im_start|>assistant\n"

// llama3
"<|start_header_id|>system<|end_header_id|>\n\n%s<|eot_id|>\n<|start_header_id|>user<|end_header_id|>\n\n%s<|eot_id|>\n<|start_header_id|>assistant<|end_header_id|>\n\n"

// mistral
"[INST] <<SYS>>\n%s\n<</SYS>>\n\n%s [/INST]"
```

### 4.4 Why This Matters for v6

v6 generates explicit code for each model. If we know the model architecture (Qwen2, Llama, etc.), we can:
1. Embed the correct chat template in the generated code
2. Avoid Python preprocessing
3. Make the CLI truly standalone (no Python dependencies for chat)
